Claim Validation

Probabilistic Member Matching
Delivered

Automates matching of incoming HCFA/UB04 claims to internal members using Splink probabilistic linkage. Handles misspellings, transposed DOBs, abbreviated addresses. Applicable across 6+ use cases including duplicate detection, provider linkage, and eligibility verification.

85%+ Match Accuracy
6+ Use Cases
2 Trained Models
3-Day Service Window
Splink Double Metaphone Azure Storage Queue SQL Server

Platform & Scale

  • Event-Driven: Azure Storage Queue decouples claim intake from matching. Claims drop into the queue as they arrive; worker processes pull and match asynchronously. No blocking, no lost claims during peak volume.
  • Probabilistic Linkage: Splink uses Fellegi-Sunter model with blocking rules to avoid O(n²) comparisons. Blocking on phonetic codes + DOB year reduces candidate pairs by 95% while maintaining recall.
  • Feature Engineering: 10+ matching features including Levenshtein distance on names, Jaro-Winkler on addresses, exact match on SSN fragments, and temporal proximity on date-of-service. Each feature contributes a log-likelihood ratio.
  • Threshold Tuning: 85% match threshold calibrated via manual review of 500+ edge cases. Below-threshold matches route to a human review queue with pre-computed similarity scores for faster adjudication.
  • Scale Path: Queue-based architecture scales horizontally — add workers to handle claim volume spikes during open enrollment periods. Blocking strategy keeps matching latency under 3 seconds per claim.
  1. Claim arrives via Azure Storage Queue
  2. Reads claim details, normalizes all fields
  3. Double Metaphone encodes names phonetically
  4. Splink blocking rules + scoring on 10+ features
  5. Best match above 85% threshold selected and linked

CPT Analysis

AI Surgical Case Intelligence
POC Completed

Google for surgical procedures. When a surgery bills 8-15 CPT codes, the system identifies the primary procedure at 85-90% accuracy with a 7-factor scoring algorithm. Enables payment pattern analysis, case benchmarking, and billing optimization.

607,813 Cases Indexed
3,504 CPT Codes
85-90% Accuracy
0.1s Query Time
ChromaDB Sentence Transformers Claude MCP Streamlit GPT-4 XGBoost

Platform & Scale

  • Multi-Engine Design: 4 independent AI engines (Primary CPT ID, Pattern Recognizer, Similarity Matcher, Recommender) run in parallel. Results are combined via a weighted ensemble — any engine can fail gracefully without breaking the pipeline.
  • Vector Database: ChromaDB stores 607K+ surgical case embeddings generated by Sentence Transformers. HNSW indexing delivers 0.1s query times — 50-300x faster than brute-force SQL lookups across the same dataset.
  • Scoring Algorithm: 7-factor weighted scoring (RVU 35%, Payment 25%, Tier 15%, Service 10%, Date 8%, Modifiers 7%) identifies primary procedures. Weights were calibrated against 1,000+ manually reviewed cases with clinical input.
  • 3 Access Modes: Claude MCP server exposes 7 tools for programmatic access. Streamlit UI with GPT-4 for interactive exploration. Direct vector search API for batch processing. Same data layer, three consumption patterns.
  • Scale Path: ChromaDB supports sharding for datasets beyond 1M cases. Embedding generation is parallelizable across GPU instances. Adding new CPT codes requires only an incremental index update, not a full rebuild.
  1. 4 AI Engines: Primary CPT ID, Pattern Recognizer, Similarity Matcher, Recommender
  2. 7-factor scoring: RVU 35%, Payment 25%, Tier 15%, Service 10%, Date 8%, Modifiers 7%
  3. Pattern Recognizer identifies 15+ surgery types from CPT combinations
  4. ChromaDB + Sentence Transformers vector search (50-300x faster)
  5. 3 modes: Claude MCP (7 tools), Streamlit UI (GPT-4), Vector Search
